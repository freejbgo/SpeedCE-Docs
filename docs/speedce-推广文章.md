## 二、核心功能详解

### 2.1 三种检测协议：HTTP、HTTPS、PING

不同的问题，需要不同的检测方式。SpeedCE 在同一页面提供三种协议切换，无需跳转，一键选择。

#### HTTP 测速

适用于检测网站在 **80 端口**的 Web 服务可用性与响应速度。当你想知道「这个域名能不能被正常访问」「页面返回是否及时」时，HTTP 模式是最直接的选择。

典型场景：

- 新站上线后验证 Web 服务是否正常；
- 排查 Nginx / Apache 配置是否生效；
- 检测 CDN 回源是否成功；
- 确认 HTTP 强制跳转 HTTPS 之前的中间状态。

#### HTTPS 测速

适用于检测 **443 端口**的加密 Web 服务。相比 HTTP，HTTPS 测速会包含 SSL/TLS 握手过程，因此能更真实地反映用户通过浏览器访问你网站时的体验。

典型场景：

- 证书部署后验证全站 HTTPS 是否可用；
- 排查 SSL 配置错误导致的部分地区访问失败；
- 评估 HTTPS 对首屏时间的影响；
- 电商、支付、登录页等安全页面的可用性巡检。

#### PING 测速

适用于检测目标 IP 的 **ICMP 连通性与网络延迟**。PING 不关心 Web 服务是否配置正确，只回答一个问题：**这个 IP 从远方能不能「打得通」，延迟是多少毫秒。**

典型场景：

- 购买 VPS 后验证商家宣传的线路质量；
- 服务器禁 Ping 前的备用方案（可配合 HTTP 模式）；
- 快速判断是网络层问题还是应用层问题；
- 对裸 IP、非 Web 服务进行基础连通性测试。

**三种模式互补，构成完整的「网络层 + 应用层」检测链路。** 当你发现 PING 通畅但 HTTPS 失败，问题很可能出在 Web 服务或证书；当 PING 和 HTTPS 都失败，则更需要从网络路由、防火墙、DNS 方向排查。

## 三、谁应该使用 SpeedCE？

### 3.1 个人站长与博客作者

你一个人维护网站，没有 7×24 监控预算，但希望在改完 DNS、换完主题、迁完服务器后，能快速确认「全国各地还能不能打开」。

**推荐用法：**

1. 改 DNS 后，用 HTTPS 模式测主域名，范围选「中国节点」；
2. 看地图有没有大片红色，重点看目标用户所在的省份；
3. 若有海外读者，再切「全球节点」补测一遍。

### 3.2 VPS / 云服务器玩家

在 HostLoc、NodeLoc 等社区里，「线路好不好」是永恒话题。商家宣传 CN2 GIA、9929、CMIN2，买家将信将疑。

**推荐用法：**

1. 拿到 VPS IP 后，用 PING 模式 + 中国节点，看电信/联通/移动延迟分布；
2. 若 PING 不通（部分云厂商禁 ICMP），改用 HTTPS 模式测 80/443；
3. 截图地图发帖，比一句「我本地 ping 20ms」更有说服力。

### 3.3 企业运维与 SRE

你们有监控系统，但业务同学、合作方、客户经常问：「为什么我这里慢？」你需要一个**对外可分享、结果直观**的第三方测速链接。

**推荐用法：**

1. 故障期间，把 SpeedCE 测速结果截图附在工单回复里；
2. 新服务上线前，建立「中国 + 全球」基线数据；
3. 与 CDN 厂商扯皮时，用多省份数据说话。

### 3.4 前端与全栈开发者

你在做国际化站点，需要验证静态资源 CDN、API 网关、Serverless 函数在不同区域的响应。

**推荐用法：**

1. 分别测 `www.example.com` 与 `api.example.com`；
2. HTTP vs HTTPS 对比，排查证书或混合内容问题；
3. 全球节点模式下，关注欧美与东南亚差异，指导边缘节点部署。

### 3.5 跨境电商与外贸团队

客户说「网站打不开」，你怀疑是 GFW、DNS 污染还是服务器宕机。

**推荐用法：**

1. 先用 HTTPS 测国内节点，确认源站是否存活；
2. 再用全球节点测客户所在国（如美国、德国、阿联酋）；
3. 若国内正常、海外异常，重点查 CDN 与跨境线路；若国内也异常，先修源站。

## 四、手把手教程：如何用 SpeedCE 完成一次完整测速

### 第一步：打开官网

访问 **https://www.speedce.com**，或中文版 **https://speedce.com/?lang=zh-CN**。

### 第二步：选择测试范围

在「测试范围」处选择：

- **中国节点**——关注国内三网及各省访问质量；
- **全球节点**——关注国际访问与出海业务。

### 第三步：选择协议

根据你的目的选择：

- 测网站能不能打开 → **HTTPS**（推荐）或 HTTP；
- 测 IP 连通与延迟 → **PING**。

### 第四步：输入目标

在输入框填写域名或 IP，例如：

- `www.baidu.com`
- `api.yoursite.com`
- `203.0.113.10`

### 第五步：开始测速

点击 **「开始测速」**。页面会显示进度，如「正在检测节点 12 / 86（14%）」。

测试过程中，地图上的节点会陆续从「等待」变为「检测中」，最终变为「通畅」或「异常」。你可以随时点击 **「停止测试」** 提前结束。

### 第六步：解读结果

关注以下指标：

1. **通畅 vs 异常比例**——若异常率超过 30%，说明存在明显可用性问题；
2. **平均延迟**——结合业务类型判断，静态博客 500ms 可能尚可，API 接口则太慢；
3. **地图分布**——是「某一省全红」还是「全国随机红点」？前者像区域性故障，后者像源站不稳定；
4. **运营商差异**——切换电信/联通/移动筛选，排查是否单网故障。

### 第七步：保存与分享

截图地图和统计数据，附在：

- 工单回复；
- 团队 Slack / 飞书群；
- HostLoc 测评帖；
- 客户邮件。

SpeedCE 的结果足够直观，**非技术人员也能看懂**——这是它相比纯表格工具的一大优势。

## 七、技术视角：HTTP、HTTPS 与 PING 测速的原理与局限

### 7.1 PING（ICMP）测速

PING 使用 ICMP 协议，测量数据包往返时间（RTT）。优点是**简单、快速**；缺点是：

- 部分服务器禁用 ICMP，会显示超时，但 Web 服务仍正常；
- PING 只反映网络层，不能代表页面加载速度；
- 高延迟不一定等于「网站慢」，低延迟也不一定等于「网站快」。

**结论：** PING 适合排查「路通不通」，不适合单独评价「网站快不快」。

### 7.2 HTTP / HTTPS 测速

HTTP(S) 测速模拟用户发起 Web 请求，测量从发起连接到收到响应的时间。HTTPS 还包含 TLS 握手，更接近真实浏览器访问。

优点是**更贴近实际业务**；缺点是：

- 受服务器并发、WAF、CDN 缓存策略影响；
- 单次测速不代表长期趋势，建议多次对比；
- 动态页面与静态页面差异大，测速结果需结合页面类型解读。

**结论：** 网站可用性检测，优先 HTTPS；IP 连通性初筛，用 PING。

### 7.3 多节点测速的意义

中国互联网呈「分省分网」结构，同一网站在不同省份、不同运营商的体验可能天差地别。全球互联网则受跨境路由、国际出口带宽、当地 ISP 质量影响。

因此，**单点测速必然有偏见，多节点测速才是公平视角。** SpeedCE 的价值，正是用分布式的节点网络，帮你还原这种「分布式真实」。
